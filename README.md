# PokeyMAX MOD Player

A ProTracker 4-channel `.MOD` player for the Atari 8-bit family (800XL/XE),
using the **PokeyMAX v1.26** sample player hardware.

Written in **cc65 C** with a small 6502 assembly VBI/IRQ stub.

---

## Hardware Requirements

- Atari 800XL / 130XE (or compatible)
- **PokeyMAX** installed with sample player support (`CAPABILITY` bit 5 = SAMPLE)
- PAL or NTSC (automatically detected via `$D210` MODE register)
- Recommended: 130XE (128KB) for loading large MOD files

---

## How It Works

### Memory Map Used

```
PokeyMAX block RAM: $0000–$A7FF (42KB = 43,008 bytes)
  Divided between samples at load time (bump allocator).

PokeyMAX registers used:
  $D284–$D293  SAMPLE player (DMA, IRQ, period, volume, format)
  $D280–$D283  COVOX manual volume (silencing)
  $D20C        CONFIG unlock / ID detect
  $D210        MODE (PAL detection)
```

### Channel Mapping

| MOD Channel | PokeyMAX HW Channel | Audio Output |
|-------------|---------------------|-------------|
| 1 (left)    | Channel 1           | Left         |
| 2 (right)   | Channel 2           | Right        |
| 3 (left)    | Channel 3           | Left         |
| 4 (right)   | Channel 4           | Right        |

### Sample Formats

The loader automatically chooses:

| Condition | Format | Quality |
|-----------|--------|---------|
| Total samples ≤ 42KB | 8-bit signed | Full 8-bit |
| Total samples > 42KB | IMA ADPCM | ~12-bit (4:1 compression) |
| Individual sample ≤ 512 bytes | Always 8-bit | Full 8-bit |

ADPCM is generated by the loader at load time using an IMA ADPCM encoder
(compatible with `sox -e ima-adpcm` output that PokeyMAX expects).

### Period Conversion

Amiga Paula clock ≈ PAULA_CLOCK Hz.  
PokeyMAX core clock = 2 × φ2 ≈ PAULA_CLOCK.

Therefore:  
```
hw_period = amiga_period
```

This is exact for PAL (both clocks derive from the same crystal).

### Timing Model

Uses ProTracker BPM model:
- **Ticks per row** = speed (set by `Fx`, x < 32), default 6
- **Ticks per second** = BPM × 24 / 60
- **VBI per tick** = VBI_Hz / ticks_per_second

An 8.8 fixed-point accumulator fires ticks at the correct sub-VBI rate for
non-standard BPMs.

At 125 BPM / speed 6 / PAL 50Hz: exactly 1 VBI per tick (classic ProTracker timing).

## Effects Supported

| Code | Effect | Notes |
|------|--------|-------|
| `0xy` | Arpeggio | Approximate semitone calculation |
| `1xx` | Slide up (portamento up) | Per-tick period decrease |
| `2xx` | Slide down (portamento down) | Per-tick period increase |
| `3xx` | Tone portamento | Slide to target note |
| `4xy` | Vibrato | Sine table, speed x, depth y |
| `Cxx` | Set volume | 0–64 |
| `Dxx` | Pattern break | Jump to next order, row xx |
| `E1x` | Fine slide up | |
| `E2x` | Fine slide down | |
| `E6x` | Pattern loop | Loop within pattern |
| `E9x` | Retrigger note | Every x ticks |
| `EAx` | Fine volume up | |
| `EBx` | Fine volume down | |
| `ECx` | Note cut | Silence at tick x |
| `EDx` | Note delay | Trigger at tick x |
| `EEx` | Pattern delay | Delay row advance by x rows |
| `Fxx` | Set speed/BPM | x<32 = speed, x≥32 = BPM |

**Not yet implemented** (future work):
- `5xx` Tone portamento + volume slide
- `6xx` Vibrato + volume slide
- `7xx` Tremolo
- `8xx` Set panning (not applicable to PokeyMAX fixed stereo)
- `9xx` Set sample offset
- `Axx` Volume slide
- `Bxx` Position jump

## Building

### Prerequisites

```bash
# Install cc65
# Ubuntu/Debian:
sudo apt install cc65

# macOS (Homebrew):
brew install cc65

# Or build from source: https://github.com/cc65/cc65
```

### Compile

```bash
cd pokeymod
make
```

Output: `modplay.xex`

### Optional: Create ATR disk image

```bash
make atr    # Requires mkatr or similar tool
```

## Usage

Copy `modplay.xex` to your Atari (via SIO2PC, AtariMax, etc.) or include in an ATR image.

**Run from SpartaDOS X or MyDOS:**
```
modplay.xex myfile.mod
```

**Default (if no argument given):**
Loads `D1:MOD.DAT` from drive 1.

**Controls:**
- `SPACE` – Pause / Resume
- Any other key – Stop and exit

**Status display:**
```
Order:   3 / 16   Row: 42/64   BPM: 125  Speed: 6
```

## File Size Limits

| Machine | Available RAM | Max MOD header+patterns |
|---------|--------------|------------------------|
| 800XL (64KB) | ~40KB after OS/player | ~32KB patterns + header |
| 130XE (128KB) | ~100KB | ~64KB patterns (128 patterns max) |

Sample data is streamed to PokeyMAX RAM during load and not kept in 6502 RAM,
so sample data size is limited only by the 42KB PokeyMAX block RAM.

For very large MODs (>42KB samples), ADPCM compression is applied automatically,
giving effective capacity of ~168KB of original 8-bit sample data.

## Known Limitations / Future Work

1. **Pattern data RAM**: Large MODs with many patterns (>64) may not fit in 6502 RAM.
   Solution: stream pattern data from disk during playback (requires DOS with random seek).

2. **Sample offset effect (`9xx`)**: Not implemented. Needed by some MODs.

3. **Volume slide (`Axx`, `5xx`, `6xx`)**: Not implemented. Common effect.

4. **Position jump (`Bxx`)**: Not implemented.

5. **Sample looping ADPCM**: Loop points in ADPCM mode need careful alignment
   to nibble boundaries. Currently the loop start/end may be slightly off for
   ADPCM samples.

6. **SID / PSG mode**: The player could be extended to use SID or PSG chips for
   additional channels (e.g. 8-channel MODs or enhanced music).

7. **XEX load segment**: On a 64KB machine the `mod_buffer` (64KB+) will overflow.
   This needs a custom loader that pages the buffer through extended RAM on 130XE.

## Technical Notes

### Why IMA ADPCM?
The PokeyMAX natively decodes IMA ADPCM at ~12-bit quality, giving 4:1
compression vs 8-bit PCM. This means 42KB of block RAM can hold ~168KB of
original sample data — enough for most 4-channel MODs from the Amiga era.

### Finetune
MOD finetune values (-8 to +7) are applied using an 8.8 fixed-point ratio
table that adjusts the Amiga period value before conversion to PokeyMAX register.

### Sample loops
Sample looping is handled via the PokeyMAX sample-end IRQ. When a sample
finishes, the IRQ fires and the loop handler re-arms the channel with the
loop region address/length. Because PokeyMAX buffers these values and applies
them at sample end, the loop is seamless with no gap.
